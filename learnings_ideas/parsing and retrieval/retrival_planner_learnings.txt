RETRIEVAL PLANNER - LEARNING NOTE

==================================================
0. THE PROBLEM EXAMPLE
==================================================

User query:
    "Summarize this document"

Selected document:
    "RAG vs Long Text"

Expected behavior:
    The system should summarize the main paper:
    - compare RAG vs long-context models
    - explain the main findings
    - mention SELF-ROUTE
    - describe the performance/cost tradeoff

Bad behavior that happened:
    The answer focused on prompt tables, QMSum, benchmark prompt design,
    and appendix-like content instead of the paper’s main message.

Why this happened:
    The old retrieval pipeline treated:
        "Summarize this document"
    almost the same as any other query.

It did:
    raw query
    -> retrieve top chunks
    -> build context
    -> generate answer

The retriever only looked for chunks that matched the raw query well.
It did not understand:
    "for a summary, I should retrieve the document's main sections"

So it could accidentally pull chunks from the wrong region of the paper,
such as appendix or prompt details, and then the generator summarized
those wrong chunks faithfully.

Important lesson:
    The generator was not the main problem.
    Retrieval selected the wrong evidence first.

==================================================
1. WHY WE ADDED A PLANNER
==================================================

Before this change, the system treated most queries with one retrieval style:

    user query
    -> retriever.search(raw_query)
    -> top chunks
    -> context builder
    -> generator answer

This is acceptable for narrow questions such as:
    "What does the paper say about SELF-ROUTE?"
or
    "What datasets were used?"

But it is weak for broad requests such as:
    "Summarize this document"
or
    "What is this paper about?"

Reason:
    The retriever sees words.
    It does not automatically understand user intent.

So we added a planner.

The planner is a small decision layer before retrieval.
Its job is not to answer the question.
Its job is to decide HOW retrieval should happen.

==================================================
2. WHAT THE PLANNER DOES
==================================================

The planner reads the query and decides:

- What kind of query is this?
  - factual
  - explanation
  - broad summary
  - comparison

- Should retrieval use one query or multiple queries?

- How many chunks should be retrieved?

- What rerank instruction should be used?

- Which document sections should be preferred?
  - abstract
  - introduction
  - results
  - conclusion

- Which sections should be discouraged?
  - references
  - appendix
  - acknowledgements

Core idea:
    Different query types need different retrieval strategies.

==================================================
3. MAIN MODULES AND WHAT THEY DO
==================================================

A. GroundedAnswerService
This is the main orchestration layer.

It does:
1. receive the user query
2. retrieve chunks
3. build grounded context
4. generate the final answer

What changed:
- instead of directly using the retriever with the raw query,
  it now calls the planner first

New role:
- service coordinates the pipeline
- planner decides retrieval strategy
- retriever executes that strategy

B. QueryPlanner
This is the new module.

It does:
1. inspect the user query
2. classify the query type
3. create a retrieval plan
4. run one or more retrieval searches
5. merge duplicate hits
6. boost useful sections
7. penalize noisy sections
8. keep a diverse final set of chunks

Important:
    The planner does not generate answers.
    It improves chunk selection.

C. QdrantHybridIndex
This is the retriever.

It does:
1. dense retrieval
2. sparse retrieval
3. fuse both result sets
4. rerank fused candidates
5. blend scores
6. return final chunks

What changed:
- it can now accept a custom rerank instruction from the planner

That means:
- factual query can use factual reranking
- summary query can use summary-oriented reranking
- comparison query can use comparison-oriented reranking

D. build_grounded_context
This takes the final selected chunks and packs them into the prompt context.

We did not change this in the first step.

E. LocalQwenGenerator
This is still the answer generator.

It receives:
- query
- grounded context

and writes the final answer.

We did not change the generation logic in this step.

==================================================
4. OLD SEARCH PROCESS
==================================================

OLD FLOW

1. user sends query
2. GroundedAnswerService sends raw query directly to retriever
3. retriever runs:
   - sparse retrieval
   - dense retrieval
   - fusion
   - reranking
   - final ranking
4. top chunks are returned
5. context builder packs them
6. generator answers from that context

Problem:
    The retrieval strategy was the same for all query types.

That means:
- narrow query and broad summary query were handled too similarly
- broad summary requests often got narrow or noisy evidence

==================================================
5. NEW SEARCH PROCESS
==================================================

NEW FLOW

1. user sends query
2. GroundedAnswerService sends the query to QueryPlanner
3. QueryPlanner classifies the query type
4. QueryPlanner builds a retrieval plan
5. QueryPlanner decides:
   - one query or multiple retrieval queries
   - how many candidate chunks to fetch
   - which rerank instruction to use
   - which sections to prefer
6. QueryPlanner runs retrieval through QdrantHybridIndex
7. retriever returns candidate chunks
8. planner merges duplicate hits
9. planner rescales scores using section-aware logic
10. planner enforces diversity
11. context builder packs final chunks
12. generator produces the final answer

Core improvement:
    Retrieval is now intent-aware instead of raw-query-only.

==================================================
6. WHAT "QUERY PLANNING" REALLY MEANS
==================================================

Query planning means the system asks:

    "What is the user trying to do?"

instead of only asking:

    "Which chunks match these exact words?"

That changes retrieval a lot.

Examples:

Query:
    "What is the accuracy on dataset X?"
Meaning:
    narrow factual lookup
Action:
    focused retrieval

Query:
    "Explain the method"
Meaning:
    explanation query
Action:
    slightly broader retrieval

Query:
    "Summarize this document"
Meaning:
    broad document-level request
Action:
    multi-query retrieval + section-aware ranking + diversity control

==================================================
7. EXAMPLE 1 - FOCUSED QUERY
==================================================

User query:
    "What does the paper say about SELF-ROUTE?"

Step 1: planner reads the query
The planner sees that the user is asking about one specific concept.

Step 2: planner classifies the query
Likely class:
    factual
or
    explanation

Step 3: planner creates a plan
Possible plan:
- retrieval query 1:
    "What does the paper say about SELF-ROUTE?"
- retrieval query 2:
    "SELF-ROUTE method results"
- use factual or explanation-style reranking
- retrieve a moderate number of chunks
- keep a focused final set

Step 4: retriever runs search
For each retrieval query, QdrantHybridIndex does:
- sparse retrieval
- dense retrieval
- fusion
- reranking
- final ranking

Step 5: planner merges results
If the same chunk appears from multiple retrieval queries,
that is a useful signal that the chunk is important.

Step 6: planner keeps focused evidence
Because this is a narrow query, it is okay if most selected chunks
come from method/results/conclusion areas.
We do not need broad document coverage here.

Step 7: context builder builds prompt context
Only the most relevant chunks are packed.

Step 8: generator answers
Now the final answer stays focused on SELF-ROUTE.

Why this works:
    The system treats the query as a targeted question,
    not as a whole-document summary request.

==================================================
8. EXAMPLE 2 - BROAD SUMMARY QUERY
==================================================

User query:
    "Summarize this document"

This is the important case.

Step 1: planner reads the query
The planner sees:
- this is broad
- this is document-level
- the user wants an overview, not one fact

Step 2: planner classifies the query
Class:
    broad_summary

Step 3: planner creates multiple retrieval probes
Instead of only searching:
    "Summarize this document"

it creates broader retrieval probes such as:
- original query
- "abstract introduction overview problem statement main contribution"
- "method approach experiment results findings"
- "conclusion discussion limitations future work"

Why this matters:
    The raw query is too generic.
    These probes give retrieval better anchors into the document.

Step 4: retriever runs search for each probe
For each probe, QdrantHybridIndex does:
- sparse retrieval
- dense retrieval
- fusion
- reranking
- final ranking

Now the system has several groups of candidate chunks.

Step 5: planner merges duplicate hits
If a chunk appears under multiple retrieval probes,
that suggests it is central to the document.

Example:
A chunk from the abstract might appear in:
- overview probe
- contribution probe
- summary probe

That chunk should usually be trusted more.

Step 6: planner applies section-aware rescoring
The planner looks at metadata such as headings.

It may boost:
- abstract
- introduction
- method
- results
- conclusion

It may penalize:
- appendix
- references
- acknowledgements

Why:
    Good summaries usually come from the main paper body,
    not from end matter.

Step 7: planner applies diversity filtering
This is important.

Without diversity control, retrieval might return:
- several chunks from one appendix section
or
- several chunks from one page

The planner limits that by restricting:
- too many chunks from one heading
- too many chunks from one page

Result:
    The final chunk set is more balanced.

Step 8: context builder packs the selected chunks
Now the context is more likely to represent the whole document.

Step 9: generator answers
The generator now sees evidence from:
- the start of the paper
- core method/results
- the conclusion

So the final summary is much more faithful.

Why this works:
    The system moves from local relevance
    to intent-aware and structure-aware retrieval.

==================================================
9. WHAT CHANGED IN CODE
==================================================

In answer_service.py
We changed the retrieval entry point.

Before:
    retrieval method
    -> self._index.search(query, doc_ids=doc_ids)

After:
    retrieval method
    -> self._planner.retrieve(index=self._index, query=query, doc_ids=doc_ids)

Meaning:
    answer service no longer decides retrieval directly
    it delegates retrieval strategy to the planner

In qdrant_hybrid_index.py
We added support for:
    rerank_instruction: str | None = None

Meaning:
    the planner can tell the retriever how relevance should be judged

Examples:
- factual query:
  "does this chunk answer the question?"
- summary query:
  "is this chunk useful for a faithful summary?"
- comparison query:
  "is this chunk useful for comparing the requested items?"

==================================================
10. WHY THIS IMPROVES THE SYSTEM
==================================================

This improves the system because retrieval is no longer blind to user intent.

Before:
- all queries used almost the same retrieval behavior

After:
- retrieval strategy depends on the task

So now:
- narrow questions stay focused
- broad summary queries become broader and more structured
- chunk selection becomes more intentional
- final answers become more grounded

==================================================
11. FINAL SUMMARY
==================================================

What we did:

1. Added a QueryPlanner before retrieval
2. Let the planner classify the query type
3. Let the planner choose retrieval strategy
4. Let the planner run multiple retrieval probes when needed
5. Let the planner rescore chunks using document structure
6. Let the planner keep a more diverse final chunk set
7. Let the retriever accept task-specific rerank instructions

Core idea:

    Better answers do not start at generation.
    They start with better chunk selection.

One-line takeaway:

    We changed the system from raw-query retrieval
    to intent-aware retrieval.
