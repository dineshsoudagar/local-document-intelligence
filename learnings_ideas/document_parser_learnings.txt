Document Parsing and Chunking Learnings (Session Summary)

Goal
-----
Build a reliable local document ingestion pipeline for a RAG system. The parser must extract clean text from PDFs, preserve document structure, and produce chunks suitable for embedding and retrieval.

Key Lessons Learned
-------------------

1. Chunking Must Respect Document Structure
-------------------------------------------

Early attempts tried to chunk based only on raw text length or naive paragraph splitting.

Example of incorrect chunk:

    Inference
    Fine-tuning
    RADA
    G-Retriever
    Pre-training
    RAPTOR
    UniMS-RAG
    CRAG
    ...

This text came from a **diagram inside the PDF**, not the actual document body.
Because the parser simply read all text elements, it accidentally treated diagram labels as meaningful content.

Lesson:
Chunking cannot rely purely on text streams. The parser must understand the **document structure**.

Correction:
Use a structured document parser (Docling) that exposes document elements like:

    paragraph
    heading
    list
    table
    figure
    caption
    formula


2. Avoid Heuristic Text Filtering
---------------------------------

A quick fix was attempted using heuristics like:

    if len(words) == 1 and len(text) < 20:
        ignore

This tried to remove diagram labels.

Problem:
This would break valid documents such as resumes or skill lists.

Example of legitimate content that would be wrongly removed:

    Python
    C++
    PyTorch
    TensorRT
    Docker
    Kubernetes

Lesson:
Never remove content using word-length heuristics.
Instead, rely on **semantic structure from the document parser**.


3. Structured Parsing is Better than Plain Text Extraction
----------------------------------------------------------

Two approaches were compared:

PyMuPDF4LLM:
    - Extracts text directly from PDF streams
    - Produces reasonably clean text
    - But loses document structure

Docling:
    - Parses full document structure
    - Provides headings, lists, captions, tables, etc.
    - Preserves page provenance and layout information

Conclusion:
Docling is more powerful because it provides **semantic structure**, not just text.


4. Do Not Reimplement Chunking When the Parser Already Provides It
-------------------------------------------------------------------

An initial implementation manually grouped paragraphs into chunks.

Example manual logic:

    current_parts.append(text)
    if len(chunk) > chunk_size:
        flush_chunk()

This led to multiple problems:

- incorrect grouping
- headings separated from paragraphs
- figure labels mixed into chunks
- inconsistent chunk sizes

Correction:
Use Docling's native chunking:

    HybridChunker()

Example:

    chunker = HybridChunker()
    chunk_iter = chunker.chunk(dl_doc=document)

HybridChunker advantages:

- understands document hierarchy
- merges headings with content
- respects semantic boundaries
- produces better RAG chunks


5. Use Contextualized Chunk Text for Embeddings
-----------------------------------------------

Docling provides:

    chunk.text

but also:

    chunker.contextualize(chunk)

Example:

Raw chunk:

    Abstract - Large Language Models (LLMs) showcase impressive capabilities...

Contextualized chunk:

    Retrieval-Augmented Generation for Large Language Models: A Survey
    Abstract - Large Language Models (LLMs) showcase impressive capabilities...

Lesson:
Embedding the contextualized version improves retrieval because the chunk contains section context.


6. Preserve Page Provenance
---------------------------

Each chunk should retain page references.

Example metadata:

    page_start: 3
    page_end: 3

This enables:

- source citation
- debugging retrieval results
- UI highlighting
- traceability


7. Keep Chunk Metadata Minimal
------------------------------

Early versions stored excessive metadata such as:

    raw_text
    schema_name
    doc_items_count

These were unnecessary.

Final lean metadata:

    {
        "parser": "docling_hybrid",
        "block_type": "text",
        "headings": [...]
    }


8. Final Chunking Pipeline
--------------------------

The final design:

    PDF
      ↓
    Docling DocumentConverter
      ↓
    DoclingDocument
      ↓
    HybridChunker
      ↓
    contextualized chunk text
      ↓
    ParsedChunk objects
      ↓
    embedding + vector storage


9. Key Principles Going Forward
-------------------------------

- Use structured document parsers, not raw text extraction
- Avoid heuristic text filtering
- Prefer parser-native chunking over custom logic
- Preserve document provenance (page numbers)
- Keep chunk metadata minimal and meaningful
- Optimize chunk text for embedding quality


Result
------

The final parser produces:

- clean chunks
- heading-aware text
- correct page references
- no diagram noise
- consistent chunk boundaries

This provides a solid ingestion layer for the offline document intelligence system.
