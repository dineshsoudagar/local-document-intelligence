# Document Parsing and Chunking Learnings (Session Summary)

## Goal

Build a reliable local document ingestion pipeline for a RAG system. The parser must extract clean text from PDFs, preserve document structure, and produce chunks suitable for embedding and retrieval.

---

## Key Lessons Learned

### 1. Chunking Must Respect Document Structure

Early attempts tried to chunk based only on raw text length or naive paragraph splitting.

Example of incorrect chunk:

    Inference
    Fine-tuning
    RADA
    G-Retriever
    Pre-training
    RAPTOR
    UniMS-RAG
    CRAG
    ...

This text came from a diagram inside the PDF, not the actual document body.
Because the parser simply read all text elements, it accidentally treated diagram labels as meaningful content.

**Lesson:**
Chunking cannot rely purely on text streams. The parser must understand the document structure.

**Correction:**
Use a structured document parser like Docling that exposes document elements such as:

    paragraph
    heading
    list
    table
    figure
    caption
    formula

---

### 2. Avoid Heuristic Text Filtering

A quick fix was attempted using heuristics like:

```python
if len(words) == 1 and len(text) < 20:
    ignore
```

This tried to remove diagram labels.

Problem:
This would break valid documents such as resumes or skill lists.

Example of legitimate content that would be wrongly removed:

    Python
    C++
    PyTorch
    TensorRT
    Docker
    Kubernetes

**Lesson:**
Never remove content using word-length heuristics.
Instead, rely on semantic structure from the document parser.

---

### 3. Structured Parsing Is Better than Plain Text Extraction

Two approaches were compared.

**PyMuPDF4LLM**
- extracts text directly from PDF streams
- produces reasonably clean text
- loses document structure

**Docling**
- parses full document structure
- provides headings, lists, captions, tables, and figures
- preserves page provenance and layout information

**Conclusion:**
Docling is more powerful because it provides semantic structure, not just text.

---

### 4. Do Not Reimplement Chunking When the Parser Already Provides It

An initial implementation manually grouped paragraphs into chunks.

Example manual logic:

```python
current_parts.append(text)
if len(chunk) > chunk_size:
    flush_chunk()
```

This led to multiple problems:
- incorrect grouping
- headings separated from paragraphs
- figure labels mixed into chunks
- inconsistent chunk sizes

**Correction:**
Use Docling’s native chunking:

`HybridChunker()`

Example:

```python
chunker = HybridChunker()
chunk_iter = chunker.chunk(dl_doc=document)
```

**HybridChunker advantages**
- understands document hierarchy
- merges headings with content
- respects semantic boundaries
- produces better RAG chunks

---

### 5. Use Contextualized Chunk Text for Text Embeddings

Docling provides:

`chunk.text`

but also:

`chunker.contextualize(chunk)`

Example:

**Raw chunk**

    Abstract - Large Language Models (LLMs) showcase impressive capabilities...

**Contextualized chunk**

    Retrieval-Augmented Generation for Large Language Models: A Survey
    Abstract - Large Language Models (LLMs) showcase impressive capabilities...

**Lesson:**
Embedding the contextualized version improves retrieval because the chunk contains section context.

**Important refinement:**
This works well when using the plain text conversion path. It becomes problematic when picture/VLM descriptions are injected into the same text flow.

---

### 6. Preserve Page Provenance

Each chunk should retain page references.

Example metadata:

```json
{
  "page_start": 3,
  "page_end": 3
}
```

This enables:
- source citation
- debugging retrieval results
- UI highlighting
- traceability

---

### 7. Keep Chunk Metadata Minimal

Early versions stored excessive metadata such as:
- `raw_text`
- `schema_name`
- `doc_items_count`

These were unnecessary.

A lean metadata structure is better.

Example:

```json
{
  "parser": "docling",
  "block_type": "text",
  "headings": [...]
}
```

Add only what is directly useful for retrieval, debugging, or traceability.

---

### 8. Final Chunking Pipeline

The stable design now looks like this:

```text
PDF
  ↓
Docling text converter
  ↓
DoclingDocument
  ↓
HybridChunker
  ↓
contextualized chunk text
  ↓
ParsedChunk objects
  ↓
embedding + vector storage
```

For picture chunks, a second enriched converter is used:

```text
PDF
  ↓
Docling picture-description converter
  ↓
Picture items
  ↓
annotation-first picture chunks
  ↓
ParsedChunk objects
```

This keeps the main text chunking path stable while still allowing image understanding to be added separately.

---

### 9. Key Principles Going Forward

- use structured document parsers, not raw text extraction
- avoid heuristic text filtering
- prefer parser-native chunking over custom logic
- preserve document provenance
- keep chunk metadata minimal and meaningful
- optimize chunk text for embedding quality

## New Learnings From This Session

### 10. Stability Matters More Than Clever Serialization

A lot of effort went into trying to build a custom serializer around `chunk.meta.doc_items` to remove picture text from hybrid chunks and then reattach it cleanly.

That looked elegant in theory but introduced real problems:
- duplicate chunks
- missing chunks
- unstable serialization
- hard-to-debug logic
- chunk counts collapsing unexpectedly

**Lesson:**
If the stable approach is slightly slower but much more reliable, use it.

The current stable solution is:
- one converter for clean text chunking
- one converter for picture enrichment
- no fragile serializer hacks

That tradeoff was worth it.

---

### 11. Two-Converter Design Is Acceptable If It Preserves Correctness

The text-only conversion path was already stable and gave good chunking behavior in about 10 seconds.

The picture-enriched conversion path adds extra processing time, but it isolates image annotation extraction from the main text chunking flow.

**Lesson:**
Running two conversions is acceptable when it gives:
- stable text chunks
- correct structure
- isolated picture extraction
- simpler code
- less debugging pain

This is better than fighting serializer edge cases for inline figure descriptions.

---

### 12. Inline Picture Descriptions Can Corrupt Chunk Semantics

When picture description is enabled in the same conversion used for text chunking, image annotations can be injected into chunk text through contextualization.

This can break the logical flow of the chunk.

Example failure mode:

    <sentence start> ... <picture description> ... <sentence end>

This destroys readability and weakens embedding quality.

**Lesson:**
Picture descriptions should not be trusted as inline body text in normal text chunks.

The cleaner solution is to keep text chunking separate and only create picture chunks from the enriched conversion path.

---

### 13. `chunker.contextualize()` Is Useful, but Only on Clean Text Documents

`HybridChunker.contextualize(chunk)` is good when the underlying document is clean and does not include noisy picture/VLM insertions.

But once picture descriptions are injected into the same conversion path, contextualization can carry that noise into the final text chunk.

**Lesson:**
Use contextualization on the plain text converter output, not on the picture-enriched converter output.

---

### 14. Picture Chunks Should Be Annotation-First

Not all image-related text is equally useful.

The most important signal is usually the VLM annotation or description, not the figure caption.

Captions often repeat visible labels or generic figure summaries. Annotations usually contain the semantic interpretation needed for retrieval.

**Correction:**
Build picture chunk text as:
- annotation first
- caption second only if it adds non-duplicate value

This keeps picture chunks focused on the content that actually improves search and retrieval.

---

### 15. Deduplicate Annotation Text Before Indexing

Picture annotations can contain repeated or near-duplicate text.

If stored raw, they inflate chunk size and weaken retrieval quality.

**Lesson:**
Normalize and deduplicate annotation strings before building the picture chunk.

This improves chunk cleanliness and avoids embedding duplicate image descriptions.

---

### 16. Do Not Change Too Many Things at Once

Several debugging cycles became difficult because multiple moving parts changed together:
- chunking logic
- serialization logic
- picture handling
- deduplication
- merging
- reindexing

That made it hard to isolate where the failure came from.

**Lesson:**
When chunking breaks:
- inspect actual chunk metadata first
- stabilize one path
- then add improvements incrementally

This is especially important in document pipelines where parser output is already complex.

---

### 17. Inspect Real Chunk Metadata Before Designing Around It

Trying to guess what `meta.doc_items`, headings, captions, and picture items contain led to wasted effort.

The correct move was to inspect actual chunk output first.

**Lesson:**
Before building custom chunk logic around a parser, first dump and inspect:
- chunk count
- text previews
- `doc_items`
- item labels
- page ranges
- headings

Do not build assumptions on undocumented object structure.

---

### 18. Token Warnings Do Not Automatically Mean the Chunker Failed

During inspection, warnings like this appeared:

    Token indices sequence length is longer than the specified maximum sequence length for this model ...

This looked alarming, but the chunk output itself was still structurally reasonable.

**Lesson:**
Do not immediately redesign chunking because of a tokenizer warning.
Inspect actual chunk text, chunk count, and token counts first.

Only change the pipeline if the warning corresponds to real bad output.

---

### 19. Some Extraction Imperfections Must Be Tolerated Initially

Examples like:

    l, the generation and the augmentation tec

show that some chunks may still contain mid-word or boundary artifacts.

That is not ideal, but it is a smaller problem than corrupting whole chunks with diagram text or unstable custom serialization.

**Lesson:**
Solve the high-impact correctness problems first:
- clean structure
- stable chunk count
- correct provenance
- separation of text and picture processing

Small extraction artifacts can be improved later.

---

### 20. Prefer Proven Stable Code Over Premature Optimization

There was a temptation to further optimize the picture conversion path by skipping text work inside the picture converter.

But that had not been validated and would introduce assumptions about PDF content types.

**Lesson:**
Do not add untested pipeline shortcuts into the learnings or implementation.
Only capture what has been actually verified.

That is why OCR-related assumptions should not be added yet.

---

## Result

The parser now produces:
- stable text chunks from a plain structured conversion
- heading-aware chunk text
- correct page references
- separate picture chunks from enriched document conversion
- picture chunks built from annotation-first content
- fewer serializer-related bugs
- a much more reliable ingestion path for RAG

This provides a stronger ingestion layer for the offline document intelligence system.

---

## Quick Recall Summary

If you only need the short version, read this:

- Use Docling, not raw PDF text extraction, because structure matters.
- Use `HybridChunker` instead of manual chunking.
- Use contextualized text for normal text chunks, but only from the plain text conversion path.
- Do not mix picture descriptions into the same conversion used for text chunking.
- Use two converters if needed:
  - one for stable text chunking
  - one for picture enrichment
- Picture chunks should be annotation-first, with caption added only if useful.
- Preserve page provenance in every chunk.
- Keep metadata minimal and meaningful.
- Do not rely on heuristics like short-word filtering.
- Do not redesign around guesses — inspect actual chunk metadata first.
- Prefer the stable, understandable pipeline over a more clever but brittle serializer.
