1. What I built today

A. Document registry
- Added a SQLite-based document registry.
- Stored:
  - doc_id
  - file_hash
  - original_filename
  - stored_path
  - parser_name
  - parser_version
  - indexed_status
  - chunk_count
  - ingested_at
  - indexed_at
  - last_error
- Purpose:
  - maintain a real document library
  - keep document metadata separate from vector search storage

B. Shared FastAPI app state
- Stored shared backend objects in app.state:
  - paths
  - document_registry
  - index_service
  - answer_service
- Accessed them through dependency functions.
- Main lesson:
  - do not create heavy services inside route functions

C. GET /documents
- Built an endpoint to list all registered documents.
- Returned response models, not raw registry objects.
- Main lesson:
  - keep internal backend objects separate from API response schemas

D. POST /documents/ingest
- Moved from stub to real ingestion flow.
- Flow:
  1. validate PDF path
  2. compute file hash
  3. create stable doc_id
  4. check deduplication
  5. copy file into managed storage/documents
  6. create registry row
  7. index the document
  8. update registry status to indexed or failed
- Main lesson:
  - ingestion is a workflow, not a single function call

E. DELETE /documents/{doc_id}
- Built document deletion endpoint.
- Deletes:
  - registry metadata
  - managed copied file
- Main lesson:
  - document cleanup should not leave orphan files

F. POST /query
- Built query endpoint with modes:
  - grounded
  - chat
  - auto
- Returns:
  - answer
  - mode_used
  - fallback_reason
  - sources
  - used_context_tokens
  - retrieved_chunk_count
  - timings
- Main lesson:
  - route stays thin, answer service does the main work

G. Single-document query filtering
- Added optional doc_ids to query input.
- If doc_ids are provided, retrieval is limited to those documents only.
- Important fix:
  - filter had to be applied inside each Qdrant Prefetch stage
  - applying it only at the top level was not enough
- Main lesson:
  - filtering must happen at the retrieval stage where candidates are collected

H. Managed storage
- Instead of trusting external file paths, ingested files are copied into app-managed storage.
- Main lesson:
  - product code should own document files instead of depending on the user’s original file path

I. Root route
- Added GET /
- Purpose:
  - quick sanity check that API is running

2. Important architecture lessons

A. Registry and vector index are different
- Registry = document-level truth
- Qdrant = chunk-level retrieval truth
- They solve different problems

B. Route layer should stay thin
Route layer should do:
- request validation
- call service
- return response model

It should not do:
- raw SQL everywhere
- indexing logic
- filesystem management in a messy way

C. One stable doc_id matters
- registry, index, retrieval, and filtering must all use the same doc_id
- if different layers generate different ids, filtering and deletion break

D. Managed storage is necessary
- storing only source_path is fragile
- user can move or rename files
- managed local copy makes the application reliable

E. API models should be separate from internal objects
- registry records are internal
- response models define the public API contract

4. Problems faced and what they meant

A. 404 on /
- Cause:
  - no route existed for /
- Fix:
  - add GET /

B. Invalid JSON error
- Cause:
  - trailing comma in JSON request body
- Fix:
  - correct the JSON
- Lesson:
  - parser errors happen before model validation

C. Qdrant lock error with uvicorn reload
- Cause:
  - heavy services were initialized too early
  - reload can create multiple processes
  - local Qdrant does not like concurrent access to same storage
- Lesson:
  - resource lifecycle matters
  - heavy services should be created at startup/lifespan, not during import

D. Registry and Qdrant mismatch
- Cause:
  - old indexed chunks existed outside current FastAPI registry flow
- Fix:
  - clear both documents.db and Qdrant storage for a clean reset
- Lesson:
  - registry is the source of truth for the document library

8. Interview-style Q and A

Q1. What problem does your application solve?
A:
It lets users build a private local document knowledge base, index documents, and ask grounded questions over them without relying on cloud services.

Q2. Why did you choose FastAPI?
A:
FastAPI is simple, typed, and works well for structured APIs. It also makes dependency injection and request/response validation straightforward, which helped keep the backend clean.

Q3. Why did you add a separate document registry instead of using only Qdrant?
A:
Qdrant is for chunk retrieval, not document lifecycle management. I needed a persistent document catalog with status, filenames, timestamps, and errors, so I added a SQLite registry.

Q4. How does document ingestion work in your backend?
A:
The backend validates the PDF path, computes a stable hash-based doc_id, checks deduplication, copies the file into managed storage, creates a registry record, indexes the file, and updates the status.

Q5. Why is stable doc_id important?
A:
Because the same identity must be shared across registry, indexing, retrieval, deletion, and filtering. If different parts use different ids, the system becomes inconsistent.

Q6. Why did you move files into managed storage?
A:
Because external file paths are fragile. Users can move or rename original files. Managed storage makes the system own its copies and keeps the backend stable.

Q7. How did you structure the FastAPI backend?
A:
I separated it into routes, request/response models, backend services, and persistence logic. That keeps route functions thin and avoids mixing HTTP logic with indexing or database details.

Q8. How did you use dependency injection in FastAPI?
A:
I used Depends(...) to inject shared objects like the document registry, app paths, index service, and answer service into routes instead of creating them inside route functions.

Q9. How does single-document querying work?
A:
The query request accepts optional doc_ids. If doc_ids are provided, retrieval is filtered so only chunks from those documents are considered.

Q10. What bug did you face while implementing single-document filtering?
A:
At first, filtering did not work because I applied the filter only at the top level. In hybrid retrieval, I had to apply the filter inside each Qdrant prefetch stage as well.

Q11. What does your query endpoint return?
A:
It returns the answer, selected mode, fallback reason if any, source references, retrieved chunk count, context token usage, and timing information.

Q12. What issue did you face with local Qdrant during development?
A:
I hit file lock problems with uvicorn reload because heavy services were being initialized too early and local Qdrant does not allow concurrent access to the same storage folder.
