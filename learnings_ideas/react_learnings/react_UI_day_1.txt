What I learned technically:
- FastAPI is not a browser thing; it is a Python web server exposing local HTTP endpoints
- React is not a language; it is a UI library used with JavaScript or TypeScript
- React can be used to build web applications that talk to a Python backend over HTTP
- Tauri does not replace React or FastAPI; it can later package the React frontend as a desktop app
- packaging with Tauri is a later distribution concern, not something that forces a rewrite now
- frontend and backend should stay separated by a clean HTTP boundary
- in React, state is what lets the UI remember user choices and data
- useEffect is used to run logic like loading documents when the component starts
- useState stores values like documents, selected document, query text, answer, and loading state
- fetch is how the frontend calls backend endpoints
- React only shows fields that I explicitly render; backend data existing is not enough
- the key prop in a list is mainly for React’s internal list update logic
- a ternary expression is just a short form of if/else
- for this project, readability matters more than tiny performance differences between ternary and if/else

What I learned about product and architecture:
- it was a mistake to expose both scope and mode separately in the UI
- simplifying the UI to a single user-facing mode selector is cleaner
- internal backend models and user-facing UI controls do not have to match 1:1
- generator loading behavior matters for user experience
- preloading the model at app startup is a good tradeoff for a desktop-style local app
- original filenames are a metadata/display concern and should come from the document registry, which is the source of truth
- trying to push original_filename through the whole parser/index/retrieval pipeline right now is more work than needed for V1
- the cleaner immediate fix is to enrich query response sources using the registry and existing doc_id

What I should remember:
- build in small slices
- do not overcomplicate the UI before the flow works
- keep backend truth and frontend display concerns separate
- prefer correct simple fixes over broad structural changes when the product is still in V1

Important backend discussion:
- I thought generator model was loading on every query
- we checked backend code and found generator is lazy-loaded once, not per request
- then we preloaded the generator at FastAPI startup by creating LocalQwenGenerator in lifespan and passing it into GroundedAnswerService
- this makes startup slower but first query faster

Important backend filename discussion:
Problem:
- query response sources return source_file like doc_xxx.pdf instead of original uploaded filename
- root cause is source_file in indexed chunk metadata comes from managed stored filename, not original filename

We considered 3 options:
1. propagate original_filename through parser, chunk model, index, retrieval, and query response
2. rename stored files to original filenames
3. quick fix in query route using registry

Decision:
- broad propagation through the whole indexing pipeline felt too heavy for now
- renaming stored files directly was rejected because doc_id-based managed storage avoids collisions
- best V1 fix is to keep current index structure and add original_filename in the query route response using registry lookup

Important correction:

- so we should not parse doc_id from chunk_id string
